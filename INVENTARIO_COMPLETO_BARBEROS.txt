14. Inconsistências reais encontradas
--------------------------------------------------------------------------
[1] 6 templates órfãos em app/templates/cliente/ (base.html, dashboard.html,
    agendar.html, beneficios.html, historico.html, perfil.html) — nenhum é
    alcançável por nenhuma rota Flask (só cliente/app.html, entrar.html e
    cadastro.html são renderizados). São a versão Jinja pré-React destas
    telas; a versão viva é a SPA React em frontend/src/pages/ (Agendar.jsx,
    Dashboard.jsx, Beneficios.jsx, Historico.jsx, Perfil.jsx), compilada
    para app/static/cliente_portal/ e servida dentro do shell cliente/app.html.
    Nunca foram removidos (~1.100 linhas de HTML morto). Ver Parte F.

[2] Blueprint 'caixa' (app/routes/caixa.py) nunca registrado em create_app()
    — código morto confirmado (Parte A.2). A TELA barbeiro/caixa.html
    (368 linhas) continua sendo renderizada normalmente por
    GET /barbeiro/caixa/<int:agendamento_id> (views/auth.py:386-389) — ou
    seja, a URL abre, mas qualquer chamada de API que a tela faça para
    endpoints de caixa vai dar 404 (blueprint não montado). Tela viva
    apontando para backend morto.

[3] Feature 'pix_integrado' só é respeitada no FRONTEND (esconde 1 link de
    nav em gestor/base.html:540) — o BACKEND não valida essa feature em
    nenhum endpoint de aprovação PIX. Ver Parte B, item 5, para os outros
    3 achados semelhantes (relatorios_avancados, agendamento_login,
    fila_espera — features 100% decorativas, zero enforcement).

[4] Sidebar do gestor (gestor/base.html) só esconde/mostra 3 dos 11 links
    de feature (pix_integrado, cupons, produtos_venda) — os links de
    Planos e VIP sempre aparecem mesmo com a feature desligada; o gestor
    consegue LISTAR (não gateado) mas não CRIAR/EDITAR (403) — comportamento
    inconsistente entre "o que a tela deixa entrar" e "o que a API permite".

[5] gestor/cupons.py gateia até a LISTAGEM (GET) por feature_required('cupons')
    — diferente do padrão usado em produtos/planos/vip, onde só escrita é
    gateada. Não é um bug (é defensável — cupom desligado, nem mostra os
    que existem), mas é uma inconsistência de padrão entre módulos que vale
    a pena decidir conscientemente ao recriar as telas.

[6] FilaEspera (tabela + feature 'fila_espera' no catálogo) existe só como
    intenção — zero rotas, zero lógica. Se a tela de gestão de features
    mostra essa opção como "liga/desliga" pro super admin, ligar não muda
    nada no sistema (não há nada pra ligar).

[7] app/templates/super/features.html duplica manualmente (linhas 78-93) a
    lista/descrição/agrupamento das features que já existe em
    app/seeds.py:FEATURES — sem nenhuma sincronização automática. Uma
    feature nova authored em seeds.py não aparece organizada nessa tela até
    alguém editar o HTML também.

[8] Rotas "/pub/..." (sem /api/v1) para planos (pub_planos_bp) coexistem com
    "/api/v1/pub/..." como alias — o próprio código marca a versão antiga
    como "DEPRECATED: remover em v2, mantido para compatibilidade externa"
    (app/routes/pub/planos.py:51 e :73). Não é bug, mas é uma rota dupla a
    ter em mente ao mapear o contrato de API pro n8n (usar sempre a
    variante /api/v1/pub/...).

[9] Nenhuma inconsistência de "JS chama endpoint que não existe" foi
    encontrada nesta varredura (os bugs históricos desse tipo — vip.html
    chamando api.vip.* inexistente, barbeiro/produtos.html chamando
    api.produtos.listar() inexistente — já foram corrigidos em blocos
    anteriores da refatoração v1.1, conforme a memória do projeto). Não
    foi feita uma varredura de todo texto JS embutido em cada um dos 49
    templates linha a linha nesta sessão (seria um item à parte, MUITO
    grande) — se quiser essa auditoria específica e exaustiva, é um pedido
    separado.

15. O QUE EXISTE vs O QUE FALTA — as 3 frentes
--------------------------------------------------------------------------

FRENTE 1 — Features ativa/inativa por SEGMENTO/BARBEARIA
  EXISTE:
    - FeatureMetadata/FeatureBarbearia: toggle por BARBEARIA individual,
      funcional, com endpoint super (PUT .../features/<nome>) e tela HTML.
    - 7 de 11 features realmente aplicadas no backend (cupons, planos,
      vip_brindes, produtos_venda, comissao, notificacoes, historico_cliente
      — este último só num endpoint de exemplo).
    - Mecanismo de "feature nova ativa por padrão pra quem já existe"
      (ativo_por_padrao) — reaproveitável como molde para features por
      segmento.
    - Segmento/SegmentoRotulo funcionando para VOCABULÁRIO (rótulos), 4
      segmentos seedados.
  FALTA:
    - Qualquer relação entre Segmento e Feature (tabela nova necessária).
    - Enforcement real de 4 features (relatorios_avancados, agendamento_login,
      fila_espera, pix_integrado) — hoje são decoração ou parcialmente
      wired só no frontend.
    - Endpoint de features para o CLIENTE (não existe).
    - Sincronizar a sidebar do gestor para esconder TODOS os links de
      feature desligada (hoje só 3 de 11).
    - Decisão de produto: mudar segmento de uma barbearia já existente
      deveria reajustar features? (hoje não faz nada com FeatureBarbearia)

FRENTE 2 — n8n/webhooks configurável via HTML
  EXISTE:
    - Arquitetura de notificação multi-canal já pronta para extensão
      (_DESPACHANTES em app/utils/notificacoes.py) — webhook seria só mais
      uma entrada no dict.
    - Pontos de evento identificados e documentados (Parte D.8) para os 5
      eventos de negócio pedidos (criado, aprovado, cancelado, plano
      ativado, venda concluída) — 4 deles ainda sem NENHUM disparo de
      notificação hoje (só "criado" e "transferido" disparam).
    - Padrão de configuração via env var (RL_*, CLOUDINARY_*) que poderia
      inspirar variáveis N8N_* globais — mas isso não serve se cada
      barbearia precisar de uma URL própria.
  FALTA (tudo — esta frente não tem NADA construído hoje):
    - Model de configuração de webhook por barbearia (URL, secret/HMAC,
      quais eventos disparar, ativo/inativo).
    - Tela HTML (super ou gestor?) para o dono configurar isso —
      "configurável via HTML" como pedido.
    - Função `_despachar_webhook(notif)` real (requests.post com timeout,
      assinatura HMAC do payload, log de sucesso/falha).
    - Chamada de criar_notificacao(canal='webhook', ...) nos 4 pontos de
      evento que hoje não notificam ninguém (aprovação PIX, cancelamento,
      plano ativado, venda concluída) — E decidir se esses eventos também
      devem gerar Notificacao in_app, ou só webhook.
    - Fila/retry para entregas que falharem (hoje TUDO é síncrono no
      request — um webhook lento ou fora do ar atrasaria a resposta HTTP
      ao usuário final se implementado ingenuamente dentro do mesmo request).
    - Endpoint(s) API para o super/gestor gerenciar a config de webhook
      (CRUD), consumidos pela tela HTML nova.

FRENTE 3 — Modelagem e recriação de TODAS as páginas frontend
  EXISTE:
    - Inventário completo dos 49 templates Jinja + 6 páginas React, com
      linha de contagem, rota que renderiza, e perfil de acesso (Parte E).
    - Mapa completo das 217 rotas de API que o frontend consome (Parte E.13)
      — o contrato real, extraído ao vivo do servidor, não de leitura manual.
    - 2 clientes HTTP JS diferentes e documentados (Bos para server-
      rendered, api para a SPA React) com seus padrões de URL.
  FALTA:
    - Decisão de arquitetura: recriar tudo como SPA React (como já foi
      feito para o cliente) ou manter o padrão server-rendered + Bos.js
      para gestor/barbeiro/super? Hoje o projeto tem DOIS padrões
      coexistindo — decidir antes de recriar evita um 3º padrão.
    - Remoção (ou reaproveitamento consciente) dos 6 templates órfãos de
      cliente/ antes de "recriar" — hoje eles confundem qualquer inventário
      automático de "quais telas existem".
    - Se a recriação for por SPA: replicar o mecanismo de proteção por
      sessão/JWT que session_required()/cliente_session_required() dão
      hoje nas páginas server-rendered.
