2. PROVA DE EXPLAIN (Seq Scan → Index Scan)
----------------------------------------------
As tabelas `clientes` (10 linhas) e `auditoria_log` (0 linhas) no banco local
são pequenas demais para o planner considerar um índice em qualquer cenário
(Postgres sempre prefere Seq Scan em tabelas que cabem em poucas páginas,
independente de índice existir). Para produzir uma prova real, foi necessário
popular dados sintéticos temporários: 1 barbearia + 50.000 usuários + 50.000
clientes (para a query de login por usuario_id) e 30 barbearias + 90.000
linhas de auditoria_log distribuídas em lotes de 3.000 por barbearia (para a
query de auditoria por barbearia_id, ~3,3% de seletividade). Todos os dados
sintéticos foram apagados ao final (ver seção 4) — a prova ficou registrada
aqui como evidência congelada.

Metodologia: `flask db downgrade` (remove os 13 índices) → seed sintético →
`ANALYZE` → EXPLAIN ANALYZE ("antes") → `flask db upgrade` (recria os
índices) → `ANALYZE` → EXPLAIN ANALYZE ("depois") → limpeza dos dados.

── Query 1: login de cliente por usuario_id ──────────────────────────────
  SELECT * FROM clientes WHERE usuario_id = 190;

  ANTES (sem índice):
    Seq Scan on clientes  (cost=0.00..1243.12 rows=1 width=643)
      (actual time=0.047..8.991 rows=1.00 loops=1)
      Filter: (usuario_id = 190)
      Rows Removed by Filter: 50009
    Execution Time: 9.173 ms

  DEPOIS (com ix_clientes_usuario_id):
    Index Scan using ix_clientes_usuario_id on clientes
      (cost=0.29..8.31 rows=1 width=645) (actual time=0.060..0.060 rows=1.00 loops=1)
      Index Cond: (usuario_id = 190)
    Execution Time: 0.183 ms

  Redução de custo estimado: 1243 → 8.3 (~150x). Tempo real: 9.17ms → 0.18ms
  (~98% mais rápido).

── Query 2: auditoria por barbearia_id ───────────────────────────────────
  SELECT * FROM auditoria_log WHERE barbearia_id = 27;

  ANTES (sem índice):
    Seq Scan on auditoria_log  (cost=0.00..1967.00 rows=3009 width=54)
      (actual time=0.057..14.873 rows=3000.00 loops=1)
      Filter: (barbearia_id = 27)
      Rows Removed by Filter: 87000
    Execution Time: 15.232 ms

  DEPOIS (com ix_auditoria_log_barbearia_id):
    Bitmap Heap Scan on auditoria_log  (cost=34.61..912.61 rows=2880 width=54)
      (actual time=0.855..3.945 rows=3000.00 loops=1)
      Recheck Cond: (barbearia_id = 27)
      ->  Bitmap Index Scan on ix_auditoria_log_barbearia_id
            (cost=0.00..33.89 rows=2880 width=0) (actual time=0.444..0.444 rows=3000.00 loops=1)
            Index Cond: (barbearia_id = 27)
    Execution Time: 4.649 ms

  O planner escolheu Bitmap Heap Scan + Bitmap Index Scan em vez de um Index
  Scan simples — comportamento esperado do Postgres quando a seletividade é
  moderada (~3,3%, ~3000 de 90000 linhas): ainda É uso do índice novo
  (ix_auditoria_log_barbearia_id aparece explicitamente no plano), só que via
  uma estratégia de bitmap em vez de index scan direto. Não é Seq Scan.
  Redução de custo estimado: 1967 → 912 (~2,2x). Tempo real: 15.2ms → 4.6ms
  (~70% mais rápido).
