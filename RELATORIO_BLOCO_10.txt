6. SLOTS QUE EXPLICAM (BUG#2) — GET /pub/<slug>/barbeiros/<id>/slots
───────────────────────────────────────────────────────────────────────────────
Adicionadas checagens ANTES de chamar `gerar_slots`: sem `ConfiguracaoAgenda`
→ 200 com `indisponivel: true` + motivo "ainda não configurou seus
horários"; `loja_aberta=False` → motivo "não está atendendo no momento";
dia inteiro coberto por um `HorarioBloqueado` → motivo = o `motivo` do
bloqueio (ou texto genérico). Sempre 200 (nunca 503), para não quebrar
consumidores externos futuros.
**Desvio do formato pedido pela tarefa**: o exemplo da tarefa usava a chave
`"horarios"`, mas o endpoint já usa `"slots"` e o frontend (`booking.html`
e o SPA React) já consome `r.slots` — troquei o nome teria quebrado
consumidores existentes sem necessidade. Mantive `"slots"` e só ACRESCENTEI
`indisponivel`/`motivo` como campos novos.

FRONTEND (item 6):
- `booking.html`: quando `slots` vem vazio, exibe `r.motivo` (se
  `indisponivel`) no lugar do texto genérico "Tente outro dia". Validado
  com Playwright — o DOM mostra o motivo real.
- `frontend/src/pages/Agendar.jsx` (fluxo cliente autenticado): **achado
  um bug pré-existente não relacionado a este bloco** — o código fazia
  `setSlots(Array.isArray(r) ? r : [])`, mas a resposta do endpoint sempre
  foi um OBJETO (`{slots, indisponivel, motivo, ...}`), nunca um array.
  Ou seja, a grade de horários dessa tela NUNCA mostrava nada, para
  nenhum barbeiro, em nenhuma circunstância — bug de longa data, não
  causado por este script. Corrigido para `Array.isArray(r?.slots) ?
  r.slots : []` + exibição do `motivo` — sem essa correção, a validação do
  item 6 no fluxo do cliente autenticado seria impossível de verificar
  (a grade sempre apareceria vazia, com ou sem minha mudança). Bundle
  recompilado com `npm run build`.

7. GUARD DE DESATIVAÇÃO DE BARBEIRO (EC07) — gestor/profissionais.py `editar_barbeiro`
───────────────────────────────────────────────────────────────────────────────
Antes de aplicar `ativo=false`, conta `Agendamento` futuros
(`data_hora > agora`) com status em `('agendado', 'aguardando_comprovante',
'aguardando_aprovacao')` para aquele barbeiro; se houver, 409 com a
contagem exata na mensagem. A checagem acontece ANTES de qualquer mutação
no objeto (nome/bio/comissão), então uma tentativa de desativação bloqueada
não aplica parcialmente outras mudanças enviadas no mesmo request.
`gestor/barbeiros.html` já tinha `catch(e) => toast(e.message, 'error')` no
botão de desativar — nenhuma mudança de frontend foi necessária; validado
com Playwright que o toast aparece com a mensagem exata.
Desativar um barbeiro SEM agendamentos futuros continua funcionando
normalmente (regressão testada).
