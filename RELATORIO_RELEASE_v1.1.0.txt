2. AS 2 FALHAS ENCONTRADAS E CORRIGIDAS (bugs reais, não falso-positivo)
------------------------------------------------------------------------------

FALHA #1 — Estoque inicial duplicado ao criar produto
  Arquivo: app/routes/gestor/catalogo.py, função criar_produto (~linha 310)
  O quê: ao criar um produto com quantidade_estoque=10, o produto nasce no
  banco já com quantidade_estoque=10 (valor do construtor) E, em seguida, o
  código chama `estoque_service.registrar_entrada(p.id, bid, qtd, ...)` —
  que faz `UPDATE produtos SET quantidade_estoque = quantidade_estoque + 10`
  por cima do valor que já era 10. Resultado: o produto fica com 20 unidades
  em vez de 10. Reproduzido de forma determinística neste bloco (2 tenants
  QA diferentes, mesmo resultado: 10 informados → 20 gravados).
  Por que é sério: todo produto cadastrado com estoque inicial > 0 desde o
  Script 18 (produção real ainda não rodou, mas qualquer barbearia de teste
  que já tenha cadastrado produtos tem esse número dobrado no banco).
  Causa raiz: o construtor de `Produto` já grava `quantidade_estoque=qtd`
  E o serviço central de estoque (`registrar_entrada`) SOMA de novo por cima
  — dupla contagem. A correção natural é ou (a) criar o Produto com
  `quantidade_estoque=0` e deixar só o `registrar_entrada` fixar o valor (e
  gerar o registro de auditoria em `movimentacao_estoque`, que hoje também
  não existe pro estoque inicial de forma correta), ou (b) remover a chamada
  a `registrar_entrada` e criar a `MovimentacaoEstoque` inicial manualmente.
  CORRIGIDO (opção a): `Produto` agora nasce com `quantidade_estoque=0`;
  quando `qtd > 0`, só `registrar_entrada()` grava o valor real (e o
  `MovimentacaoEstoque` de auditoria correspondente, que já existia).
  Revalidado: produto criado com `quantidade_estoque=10` agora grava
  exatamente 10 (antes: 20).

FALHA #2 — Criar produto e ajustar estoque não respeitavam a feature
"produtos_venda" desligada
  Arquivo: app/routes/gestor/catalogo.py, funções criar_produto (~linha 270)
  e ajustar_estoque (~linha 414)
  O quê: nenhuma das duas tinha o decorator `@feature_required
  ('produtos_venda')` — ao contrário de `criar_categoria_produto`,
  `movimentar_estoque` (gestor/estoque.py) e `criar_venda`/
  `criar_venda_barbeiro` (todos já tinham o decorator). Testado neste
  bloco: com a feature desligada, ambos os endpoints funcionavam
  normalmente (201/200) em vez de 403, igual aos outros 3 endpoints de
  escrita do mesmo módulo.
  Por que era sério: uma barbearia sem a feature "produtos_venda"
  contratada ainda conseguia cadastrar produtos e ajustar estoque via API
  (a tela do frontend provavelmente esconde o botão, mas a API não
  bloqueava).
  CORRIGIDO: `@feature_required('produtos_venda')` adicionado aos dois
  endpoints, na mesma posição usada pelos demais (entre `@gestor_required`
  e a função). Revalidado: com a feature desligada, ambos retornam 403;
  religada a feature, voltam a funcionar normalmente (ajustar-estoque
  aplicado com sucesso, quantidade final correta).

Nenhuma outra falha foi encontrada nos 96 itens restantes. Após os 2 fixes
acima, revalidação focada (5 checagens) + smoke geral (217 rotas, `/entrar`
e `/health` 200) confirmaram tudo OK — nenhum dado de teste residual ficou

3. CONCORRÊNCIA
--------------------
Executado com servidor HTTP real (`werkzeug.serving.make_server`, não
test_client) — necessário para ter threads/conexões de verdade disputando o
mesmo lock de banco.

[1] Double-booking (teste_concorrencia.py, criado no Script 07) — 10 threads
    tentando criar o MESMO agendamento simultaneamente:
      Resultado: 1x 201, 8x 409, 1x 422 — zero double-booking.
      O script original (Script 07) espera estritamente 1x201 + 9x409, e
      por isso reporta "FALHOU". Investigado: o único código 422 observado
      vem da camada de revalidação via `gerar_slots()` adicionada no Script
      08 (POSTERIOR à escrita deste teste) — uma segunda linha de defesa que
      não existia quando a asserção foi escrita. Sob concorrência real, uma
      thread mais lenta pode ver o slot já ocupado nessa revalidação (leitura
      simples, sem lock) ANTES de chegar no INSERT que dispararia o
      IntegrityError/409. Nenhum double-booking ocorreu (exatamente 1x201) —
      o invariante que realmente importa se manteve. A asserção do script
      está desatualizada, não o código do produto. Não alterei o script
      (fora do escopo deste bloco); documentando aqui para não ser
      interpretado como regressão de concorrência.

[2] Dupla aprovação de solicitação de plano (mesmo script) — 5 threads
    aprovando a MESMA solicitação: 1x 200, 4x 409. PASSOU (sem alteração).

[3] Venda simultânea de produto (Script 18/Bloco 6.4) — teste novo escrito
    neste bloco (não existia commitado; o teste original foi descartável e
    apagado ao final do Script 18). 10 threads vendendo 1 unidade cada de um
    produto com estoque=5:
      Resultado: 5x 201, 5x 422 ("Estoque insuficiente"), estoque final no
      banco = 0 (nunca negativo). PASSOU — o lock atômico de
      `app/utils/estoque.py:registrar_saida` segura corretamente a corrida
      mesmo com produção real de estoque via SQL direto (usado no setup do
      teste para não passar pelo bug da FALHA #1 acima e não contaminar o
      resultado deste teste específico).

Ambos os testes de concorrência criaram e limparam seus próprios dados
(tenants `qa-concorrencia`/`qa-concorrencia-estoque`) — confirmado sem
resíduo no banco ao final.
