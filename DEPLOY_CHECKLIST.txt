1. VARIÁVEIS DE AMBIENTE A SETAR NO RAILWAY
------------------------------------------------
  DATABASE_URL              → Railway gera automaticamente ao provisionar o
                               plugin PostgreSQL (não copiar do .env local,
                               que aponta pro Postgres local de dev).
  SECRET_KEY                → gerar um valor NOVO e DIFERENTE do usado em
                               dev local: python -c "import secrets;
                               print(secrets.token_hex(32))"
  JWT_SECRET_KEY             → idem, outro valor novo e diferente.
  FLASK_ENV=production
  CLOUDINARY_CLOUD_NAME      → mesmo valor (não é secreto)
  CLOUDINARY_API_KEY         → mesmo valor (não é secreto sozinho)
  CLOUDINARY_API_SECRET      → o NOVO valor gerado no passo 0 acima (rotacionado)
  ADMIN_EMAIL                → e-mail real do super_admin de produção
  ADMIN_SENHA                → deixar SEM DEFINIR na 1ª execução do
                               seeds_prod.py — o script gera uma senha
                               aleatória forte e imprime UMA VEZ no log; depois
                               trocar a senha pelo próprio painel e então, se
                               quiser, fixar ADMIN_SENHA pra próximos resets.
  RATELIMIT_STORAGE_URI      → Redis do Railway se disponível (ver alerta na
                               seção 3) — se não tiver, deixar memory:// e
                               aceitar a limitação documentada.

  Variáveis com default sensato no código, só sobrescrever se precisar:
  RATELIMIT_DEFAULT, RL_LOGIN, RL_CADASTRO, RL_AGENDAR, RL_PLANO_SOLICITAR,

3. RATE LIMITING — ATENÇÃO AO NÚMERO DE WORKERS (DT-006)
-------------------------------------------------------------
  Confirmado no `Procfile` atual: `gunicorn -w 2` (2 workers). Rate limiting
  (Script 13) usa `RATELIMIT_STORAGE_URI=memory://` por padrão, que é
  POR-WORKER — com 2 workers, um mesmo IP pode efetivamente conseguir até
  ~2x o limite configurado (ex: RL_LOGIN=5/min pode, na prática, virar
  ~10/min, dependendo de qual worker atende cada request).
  Antes de considerar o rate limiting confiável em produção: provisionar
  Redis (plugin do Railway) e setar RATELIMIT_STORAGE_URI=redis://...
  Sem isso, documentar como limitação conhecida (não bloqueia o deploy,
  mas reduz a eficácia do Bloco 4.2).


4. VALIDAÇÃO ANTI-PLACEHOLDER (Script 19)
----------------------------------------------
  `app/__init__.py` agora recusa iniciar (`RuntimeError`) se SECRET_KEY ou
  JWT_SECRET_KEY:
    - tiverem menos de 32 caracteres, OU
    - contiverem um trecho de placeholder conhecido (your-secret-key,
      dev-secret, test-secret, substitua, change-me, changeme,
      secret-key-here — case-insensitive), OU
    - tiverem entropia baixa demais (um único caractere repetido > 50% do
      valor).
  Isso significa que o deploy vai FALHAR imediatamente (antes de aceitar
  qualquer request) se alguém esquecer de configurar essas variáveis no
  Railway — comportamento intencional, prefere falhar rápido a rodar
  insegura.
