ANÁLISE DO ITEM 4 (ClientePlanoUso / uq_plano_uso_dia) — feita ANTES de mudar código
───────────────────────────────────────────────────────────────────────────────
Investiguei toda referência a `ClientePlanoUso` e `semana_do_mes` no código:
- Toda verificação de limite (`_resolver_plano` em pub/agendamento.py,
  equivalente em cliente/planos.py) conta uso POR MÊS (ano+mês) — nunca por
  dia. Não existe hoje nenhuma regra de negócio de "1 uso por dia" explícita
  em lugar nenhum.
- `ClientePlanoUso.semana_do_mes` é GRAVADO mas nunca LIDO em lugar nenhum do
  código — resquício de um design que parece ter previsto granularidade
  semanal e nunca chegou a ser implementado na verificação. O campo `usado`
  (sempre True) também nunca é consultado.
- Conclusão levada para você: a constraint `uq_plano_uso_dia` (cliente_plano_id,
  servico_id, data_uso) do Bloco 2.1 é razoável para o caso de uso real — nada
  no código sugere necessidade de negócio de "2x o mesmo serviço, mesmo plano,
  mesmo dia".
- Você escolheu: dentro do MESMO booking, só a 1ª ocorrência de
  (cliente_plano_id, servico_id) consome o plano; repetições no mesmo
  pedido/dia são cobradas como avulso. Sem migration nova.

Implementado exatamente assim em `_criar_agendamento_core`
(app/routes/pub/agendamento.py) — um `set()` local rastreia (cliente_plano_id,
servico_id) já consumidos NESTE booking; se `_resolver_plano` disser que o
item é coberto pelo plano mas a combinação já foi usada neste mesmo pedido,
o item vira avulso (preço cheio). Validado: um booking com o mesmo serviço
2x retornou o 1º item com `is_plano:true, preco:0.0` e o 2º com
`is_plano:false, preco:40.0` — só 1 linha em `cliente_plano_uso` foi criada
(conferido direto no banco).

**Achado colateral, não relacionado a este bloco**: nem `pub/agendamento.py`
nem `cliente/agendamento.py` de fato repassam `cupom_codigo` do request para
`_criar_agendamento_core` — o parâmetro existe na função mas nenhum dos dois
endpoints o preenche a partir de `request.get_json()`. Ou seja, hoje não dá
pra aplicar cupom no ato de criar o agendamento via API pública/cliente
(a lógica de `validar_cupom`/`incrementar_uso_cupom` só é exercitada pelos
callers em barbeiro/gestor, que operam sobre agendamento já criado). Não
mexi nisso — é uma lacuna de funcionalidade pré-existente, fora do escopo de
concorrência deste bloco.

MUDANÇAS POR ITEM
───────────────────────────────────────────────────────────────────────────────
1. Double-booking (app/utils/agenda.py, app/routes/pub/agendamento.py):
   - `verificar_conflito`: query agora usa `.with_for_update()` nos
     agendamentos do barbeiro no dia antes de checar sobreposição.
   - `_criar_agendamento_core`: `db.session.add(ag); db.session.flush()`
     agora dentro de try/except IntegrityError → rollback + APIError
     'Este horário acabou de ser reservado. Escolha outro.' (409). Cobre o
     caso em que não há linha nenhuma pra travar (1º agendamento do
     barbeiro no dia) e a `uq_ag_barbeiro_slot` (Bloco 2.1) é quem pega.
   - `cliente/agendamento.py` usa a MESMA `_criar_agendamento_core` — corrigido
     automaticamente, sem duplicar código.

2. Dupla aprovação de plano PIX (gestor/planos.py):
   - `aprovar_solicitacao`: busca com `.with_for_update()`; status != 'pendente'
     agora é 409 (era 400); `ClientePlano.solicitacao_id=sol.id` preenchido;
     commit com except IntegrityError específico (UNIQUE de solicitacao_id)
     → 'Solicitação já aprovada por outro gestor.' (409).
   - `rejeitar_solicitacao`: mesmo lock + status 409.

3. Cupom atômico (app/utils/cupons.py):
   - `incrementar_uso_cupom`/`decrementar_uso_cupom` agora fazem UPDATE
     atômico via SQL direto (`UPDATE cupons SET quantidade_usos = ... WHERE
     ... AND (quantidade_maxima_usos IS NULL OR quantidade_usos <
     quantidade_maxima_usos)`), confirmando nomes reais de tabela/coluna
     (`cupons`, `barbearia_id`, `quantidade_usos`, `quantidade_maxima_usos`).
     `rowcount == 0` no incremento → APIError 'Cupom esgotado.' (422).
   - Mudança de assinatura: as duas funções agora exigem `barbearia_id`
     (usado no WHERE, defesa em profundidade — `cupom_id` já é PK única, mas
     segue o padrão pedido). Atualizados os 5 call sites (barbeiro/
     agendamentos.py x2, cliente/agendamento.py, gestor/agendamento.py x2,
     pub/agendamento.py x1).
   - `validar_cupom` mantido como pré-check de UX (mensagens específicas);
     a fonte da verdade agora é sempre o UPDATE atômico.

4. Limite de plano em multi-serviço: ver análise acima.

5. Dupla aprovação de comprovante PIX (agendamento):
   - `gestor/agendamento.py aprovar_agendamento` e
     `barbeiro/agendamentos.py aprovar_comprovante`: busca do Agendamento
     agora com `.with_for_update()`; mensagem de status inválido trocada
     para 'Este agendamento já foi processado.' (409, era 422/400 com
     mensagem técnica).

6. teste_concorrencia.py (novo, raiz do projeto):
   - Cria seus próprios dados de teste (barbearia/gestor/barbeiro/cliente/
     plano descartáveis), roda os 2 testes de concorrência via HTTP real
     contra um servidor de pé, imprime resumo, e apaga os dados ao final.
   - Uso: `python teste_concorrencia.py [base_url]` (default
     http://127.0.0.1:5000).
