PROBLEMAS ENCONTRADOS (fora de escopo deste bloco, registrados para depois)
───────────────────────────────────────────────────────────────────────────────
1. Drift de schema pré-existente detectado pelo autogenerate (removido da
   migration deste bloco, não tocado):
   a) agendamentos.status: DB é VARCHAR(30) (migration a3f8c2e1d047), model
      declara String(20) — model desatualizado. Se uma migration futura
      "corrigir" isso sem cuidado, vai truncar/rejeitar status como
      'aguardando_comprovante' (22 chars). Candidato a resolver no Script 03.
   b) segmentos.chave: autogenerate quis dropar a unique constraint
      'segmentos_chave_key'; parece falso positivo (model mantém unique=True).
2. auditoria_log.usuario_id não tem cascade (FK sem ON DELETE) — confirmado
   na prática durante a limpeza dos dados de QA deste teste (precisei apagar
   AuditoriaLog manualmente antes de apagar Usuario). Isso é uma instância
   NOVA do mesmo padrão já registrado como DT-002 (cascade delete em
   Agendamento → filhos) — o problema é mais amplo do que só Agendamento,
   afeta qualquer tabela de auditoria/histórico que referencia usuarios.id.
3. Observação sobre a validação "desativar barbearia → gestor recebe 403":
   na prática o gestor recebe 401 ("Autenticação necessária"), não 403. Isso
   acontece porque desativar_barbearia já fazia bulk UPDATE ativo=False em
   TODOS os usuários do tenant (comportamento pré-existente, não alterado por
   mim) — e o check `usuario.ativo` em context.py (também pré-existente) zera
   g.user_id ANTES do novo guard de tenant (403) rodar. O resultado prático é
   o mesmo (acesso bloqueado imediatamente), só o código HTTP difere do que
   o próprio texto da tarefa citou na seção de VALIDAÇÃO (a seção FLUXO 2 do
   mesmo documento já dizia "Gestor recebe 401 imediatamente" — os dois
   trechos da tarefa se contradizem entre si nesse detalhe). O guard 403 que
   implementei em decorators/auth.py continua útil como defesa em profundidade
   para o caso em que só Barbearia.ativo mude sem cascatear para Usuario.ativo.
