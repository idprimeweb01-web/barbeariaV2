REGRAS DE NEGÓCIO CRÍTICAS:
├─ 1. SLOT ÚNICO: 1 barbeiro NÃO pode ter 2 agendamentos ativos no mesmo slot
│     Status: ❌ QUEBRADA (double-booking possível em race condition)
│     Solução: índice único parcial + SELECT FOR UPDATE (Script 07)
├─ 2. LIMITE DE PLANO: cliente não pode usar mais serviços que o limite mensal
│     Status: ❌ QUEBRADA (multi-serviço passa por cima)
│     Solução: UPDATE atômico com constraint (Script 07)
├─ 3. CUPOM ÚNICO: cupom com quantidade_maxima_usos=1 pode ser usado APENAS 1x
│     Status: ❌ QUEBRADA (race condition em venda simultânea)
│     Solução: UPDATE atômico (Script 07)
├─ 4. APROVAÇÃO ÚNICA: 1 solicitação de plano = APENAS 1 ClientePlano criado
│     Status: ❌ QUEBRADA (2 gestores podem aprovar ao mesmo tempo)
│     Solução: SELECT FOR UPDATE + UNIQUE em solicitacao_id (Script 07)
├─ 5. PLANO EXPIRA: não pode usar após data_fim
│     Status: ❌ QUEBRADA (data_fim ignorada, nunca expira)
│     Solução: validação no _resolver_plano + job diário (Script 10)
├─ 6. ISOLAMENTO TENANT: gestor A não vê dados de barbearia B
│     Status: ⚠️ PARCIAL (2 IDORs via ?barbeiro_id=)
│     Solução: whitelist de filtros, validação de ownership (Script 05)
├─ 7. REVOGAÇÃO: logout = token inválido imediatamente (não 30 dias depois)
│     Status: ❌ QUEBRADA (sem blacklist, token continua válido)
│     Solução: TokenRevogado + context check (Script 01)
├─ 8. INTEGRIDADE: status, valores e percentuais não podem ser corrompidos
│     Status: ❌ QUEBRADA (validação só em Python)
│     Solução: CHECK constraints + UNIQUE no banco (Script 03)
├─ 9. TRANSFERÊNCIA: barbeiro desativado → agendamentos transferem, não cancelam
│     Status: ❌ NÃO EXISTE (novo recurso — Script 17)
│     Solução: modal de transferência, mural de agendamentos disponíveis
└─ 10. PRODUTOS: venda avulsa com estoque, movimentação auditável, comissão
Status: ❌ NÃO EXISTE (novo módulo — Script 18)
Solução: Venda, VendaItem, MovimentacaoEstoque, CategoriaProduto

CASOS EXTREMOS MAPEADOS:
├─ 2 clientes simultâneos tentam agendar no MESMO slot (race condition)
│  Antes: 2 agendamentos criados (erro)
│  Depois: 1 sucesso, 1 recebe 409 Conflict (correto)
├─ Gestor tenta desativar barbeiro com 10 agendamentos futuros
│  Antes: desativa sem aviso, clientes ficam órfãos
│  Depois: 409 "Existem N agendamentos futuros" + opção de transferência
├─ Cliente envia comprovante PIX 2x (duplo-clique)
│  Antes: sobrescreve silenciosamente
│  Depois: 422 "Comprovante já enviado" ou atualização controlada
├─ 2 gestores aprovam MESMA solicitação de plano simultaneamente
│  Antes: 2 ClientePlano criados (cliente pagou, recebeu 2 planos)
│  Depois: 1 sucesso, 1 recebe 409 "Já aprovada"
├─ Barbeiro muda horário enquanto cliente agenda o slot antigo
│  Antes: agendamento cria fora do horário comercial
│  Depois: 422 "Horário não está mais disponível"
├─ Plano expira no meio do mês (data_fim = ontem)
│  Antes: cliente continua agendando com o plano expirado
│  Depois: plano ignorado, cliente paga como avulso OU vê mensagem clara
├─ Super admin desativa barbearia
│  Antes: gestores continuam logados 30 dias (refresh token válido)
│  Depois: gestor recebe 403 imediatamente em qualquer API
└─ Barbearia sem ConfiguracaoAgenda (horário = NULL)
Antes: slots vazios sem explicação, cliente pensa que app está quebrado
Depois: 503 com mensagem "Profissional ainda não configurou horários"

DEPENDÊNCIAS ENTRE SCRIPTS:
├─ Script 03 (Constraints) é PRÉ-REQUISITO do Script 07 (Locks)
├─ Script 07 (Locks) é PRÉ-REQUISITO do Script 08 (Revalidação)
├─ Script 01 (TokenRevogado) pode ser feito independente, mas usado em Scripts 10 e 17
├─ Script 14 (Constantes) recomendado ANTES de Scripts 17 e 18
├─ Script 19 (Secrets) SEMPRE por último antes do 20 (Release)
└─ Scripts 05, 06, 09, 10 podem ser feitos em paralelo (sem bloqueadores)

FLUXO 2: LOGOUT REAL (com Revogação de Token)

ANTES (v1.0):
├─ Cliente clica "Sair"
├─ Frontend deleta JWT cookies
├─ SE atacante copiou o cookie:
│  └─ Continua válido por 30 dias (refresh token)

└─ Consegue operar por 30 dias

├─ Backend:
│  ├─ jti = get_jwt()['jti']
│  ├─ INSERT TokenRevogado(jti, usuario_id, motivo='logout')

│  ├─ unset_jwt_cookies()
│  └─ COMMIT
│
├─ Em cada request subsequente:
│  ├─ verify_jwt_in_request(optional=True)
│  ├─ Se JWT válido:
│  │  ├─ jti = get_jwt()['jti']
│  │  ├─ SELECT TokenRevogado WHERE jti = ?
│  │  ├─ Se encontra → g.user_id = None (não autenticado)
│  │  └─ Se não encontra → autorizar (JWT ainda válido)
│  └─ Se JWT inválido → não autenticado (normal)
│
└─ RESULTADO:
├─ Logout real (cookie roubado = inútil)
├─ Super admin desativa barbearia:
│  └─ Usuario.token_valido_apos = now() (revogação EM MASSA)
│  └─ Qualquer token do gestor com iat ANTERIOR = inválido
│  └─ Gestor recebe 401 imediatamente
